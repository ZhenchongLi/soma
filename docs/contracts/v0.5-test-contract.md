# v0.5 Test Contract — soma_llm_call (supervised, monitored LLM-call worker + mock provider)

This document maps each behavioural proof of the v0.5.1 `soma_llm_call` slice to
the suite and case that proves it, together with the concrete process-survival
assertion. It is the companion to
[v0.2-test-contract.md](v0.2-test-contract.md),
[v0.3-test-contract.md](v0.3-test-contract.md), and
[v0.4-test-contract.md](v0.4-test-contract.md).

Every actor-facing proof asserts **process survival**, not just a return value:
the worker runs in its own pid, the actor survives the worker's timeout / cancel
/ crash, and the actor stays responsive while a call is in flight.

## What this slice builds

A disposable worker `soma_llm_call` (in `apps/soma_runtime/src/`), shaped like
`soma_tool_call`: it runs in its own process, performs one call, sends a result
message back to its owner, then exits. The actor owns it directly — there is no
`soma_llm_call_sup`, mirroring `soma_run → soma_tool_call` (the decision locked
in [v0.4-test-contract.md](v0.4-test-contract.md)).

## Locked design decisions the proofs lean on

Settled in [../../design-77.md](../../design-77.md):

1. **Mutual exclusion** — an envelope carries `steps` (list) **or** `llm` (map),
   never both. Both present is malformed and rejected up front by
   `validate_envelope/1` with `{error, _}`, before any child starts.
2. **Owner-enforced timeout** — the actor arms a call-timeout timer when it
   starts the call and treats the timer firing as the timeout; the `slow` mock
   that ignores the timer is exactly the case this proves. Mirrors how
   `soma_run` arms the per-step `state_timeout`.
3. **Actor-driven teardown** — on timeout and cancel the actor kills the worker
   itself (`exit(WorkerPid, kill)`), because the bare worker is not a
   `gen_statem` that can drive its own teardown. The mock holds no external OS
   process, so there is nothing to reap beyond the BEAM pid.
4. **Events emitted from the actor** — `llm.started`, `llm.succeeded`,
   `llm.failed`, `llm.timeout`, `llm.cancelled` are emitted by the actor (which
   already holds the event-store handle), keeping the worker event-store-free.
   Each carries the task's `correlation_id`, so `by_correlation/2` returns them
   alongside the task's `actor.*` events.
5. **Provider seam** — the actual call lives in a single function
   `soma_llm_call:perform_call/1`, the one point a real provider grows into. The
   mock reads a `directive` (`success` / `slow` / `crash` / `hang`) out of the
   `llm` map; there is no provider behaviour or `mock` module yet.

## Proving suites

Two suites prove this slice:

- **`soma_llm_call_tests`** — an EUnit module in `apps/soma_runtime/test/`. The
  worker-only proof that touches no actor.
- **`soma_llm_call_SUITE`** — a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_actor_SUITE` (boot the `soma_runtime` app so `soma_run_sup`
  and the event store are alive, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with an `llm` envelope). Every actor-facing proof.

## v0.5.2 — proposals as validated data, not execution

The v0.5.2 slice (see [../../design-83.md](../../design-83.md)) adds a normalized
**proposal** contract: an LLM call returns a raw proposal map, which
`soma_proposal:normalize/1` turns into validated data (`reply` / `run_steps` /
`reject` / `ask`) or rejects. A proposal is *data the actor records* — a
`run_steps` proposal does **not** itself start a run. Two suites prove it:

- **`soma_proposal_tests`** — an EUnit module in `apps/soma_actor/test/`. The
  pure `soma_proposal:normalize/1` proofs, touching no actor.

  set up like `soma_llm_call_SUITE` (boot `soma_runtime`, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with a `proposal` llm directive). The actor-side proofs.

## v0.5.3 — policy gate: a proposal gets an allow/reject verdict, no execution

The v0.5.3 slice (see [../../design-85.md](../../design-85.md)) gives a normalized
proposal a **verdict**. A pure `soma_policy:check(Proposal, Policy)` returns
`allow` or `{reject, Reason}` against a tool-name allowlist
(`#{allowed_tools => [atom()] | all}`): a `run_steps` proposal is allowed only
when every step's `tool` is in the allowlist; `all` or an absent `allowed_tools`
key allows any tool; `reply` / `reject` / `ask` carry no tool and are allowed
without a check. The actor wires the gate into the existing `{proposal, Proposal}`
branch right after `proposal.created` and emits `proposal.approved` or
`proposal.rejected` accordingly — still **no execution**: an approved `run_steps`
proposal starts no run, it is recorded as approved data. Two suites prove it:

- **`soma_policy_tests`** — an EUnit module in `apps/soma_actor/test/`. The pure
  `soma_policy:check/2` proofs, touching no actor.
- **`soma_policy_SUITE`** — a Common Test suite in `apps/soma_actor/test/`, set
  up like `soma_proposal_SUITE` (boot `soma_runtime`, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with a proposal-returning `llm` directive). The actor-side gate proofs.

### Pure policy-decision proofs

| # | Proof | Suite · case |
|---|-------|--------------|
| 1 | A `run_steps` proposal whose every tool is allowed returns `allow` | `soma_policy_tests` · `run_steps_all_tools_allowed_returns_allow_test` |
| 2 | A `run_steps` proposal naming an unknown tool returns `{reject, Reason}` | `soma_policy_tests` · `run_steps_unknown_tool_returns_reject_test` |
| 3 | An `all` allowlist (or an absent `allowed_tools` key) returns `allow` | `soma_policy_tests` · `run_steps_all_or_absent_allowlist_returns_allow_test` |
| 4 | Toolless kinds (`reply` / `reject` / `ask`) return `allow` without a check | `soma_policy_tests` · `toolless_kinds_return_allow_test` |

## v0.5.4 — approved proposals execute (node A: intent → LLM → proposal → policy → run)

The v0.5.4 slice (see [../../design-87.md](../../design-87.md)) closes the loop:
an approved `run_steps` proposal now **executes**. The actor wires the run start
into the `{proposal, Proposal}` branch right after `proposal.approved` — an
approved `run_steps` proposal starts a `soma_run` under `soma_run_sup` (the actor
owns it directly, the v0.4 path), emits `proposal.executed`, and the task tracks
the run's outcome (`completed` / `failed` / `timeout` / `cancelled`); an approved
toolless `reply` proposal has nothing to run and goes straight to `completed`
carrying the normalized proposal; a policy-rejected proposal still starts no run
and rests at `rejected`. The v0.4 direct `steps` path is untouched — it runs
straight to a run and emits no `proposal.*` event. One suite proves it:

- **`soma_proposal_exec_SUITE`** — a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_policy_SUITE` (boot `soma_runtime` so `soma_run_sup` and the
  event store are alive, start an actor through `soma_actor_sup:start_actor/1`
  with a `tool_policy`, drive it through the real `soma_actor:send/2` with a
  `proposal` llm directive). The actor-side execution proofs. Each proof reads
  outcomes back through `get_task_status/2`, `get_task_result/2`, and
  `soma_event_store:by_correlation/2`.

## v0.5.5 — budget & loop limits: exhaustion fails the task, not the actor

The v0.5.5 slice (see [../../design-89.md](../../design-89.md)) gives an actor a
`budget` cap checked at its two spend points. An actor started with
`budget => #{max_llm_calls => N}` and/or `budget => #{max_steps => N}` enforces
those caps as the decision loop spends: `maybe_start_llm_call/4` fails the task
before making a call once the task's LLM-call count is already at `max_llm_calls`
(`max_llm_calls => 0` is exactly this case); the approved `run_steps` branch fails
the task before starting a run when the proposal carries more steps than
`max_steps`. Either exhaustion fails the **task** as data — recorded `failed` with
a `{budget_exceeded, max_llm_calls}` / `{budget_exceeded, max_steps}` reason
through the shared failure path — and the **actor** stays alive for the next
envelope. An absent `budget` key leaves the v0.5.4 behaviour unchanged: both
checks see an unlimited cap. One suite proves it:

- **`soma_actor_budget_SUITE`** — a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_proposal_exec_SUITE` (boot `soma_runtime` so `soma_run_sup`
  and the event store are alive, start an actor through
  `soma_actor_sup:start_actor/1` with a `budget` and a `tool_policy`, drive it
  through the real `soma_actor:send/2` / `soma_actor:ask/3`). Each proof reads
  outcomes back through `get_task_status/2` and `soma_event_store:by_correlation/2`.

### Actor-side budget proofs (criteria 1–10)

| # | Proof | Suite · case | Survival / behaviour assertion |
|---|-------|--------------|--------------------------------|
| 1 | `max_llm_calls => 0` fails the `llm` task with `{budget_exceeded, max_llm_calls}` | `soma_actor_budget_SUITE` · `budget_zero_llm_calls_fails_task_with_reason` | the task reaches `failed` with reason `{budget_exceeded, max_llm_calls}`; the actor pid stays alive |
| 2 | `max_llm_calls => 0` makes no LLM call (no `llm.started`) | `soma_actor_budget_SUITE` · `budget_zero_llm_calls_emits_no_llm_started` | `by_correlation/2` for the task carries no `llm.started` event; the actor pid stays alive |
| 3 | `max_steps => N` fails an oversized approved `run_steps` proposal with `{budget_exceeded, max_steps}` | `soma_actor_budget_SUITE` · `budget_max_steps_fails_oversized_proposal_with_reason` | a proposal over the cap reaches `failed` with reason `{budget_exceeded, max_steps}`; the actor pid stays alive |
| 4 | An oversized `max_steps` proposal starts no run (no `run.started`) | `soma_actor_budget_SUITE` · `budget_max_steps_oversized_proposal_emits_no_run_started` | `by_correlation/2` for the task carries no `run.started` event; the actor pid stays alive |
