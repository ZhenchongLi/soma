# v0.5 Test Contract ŌĆö soma_llm_call (supervised, monitored LLM-call worker + mock provider)

This document maps each behavioural proof of the v0.5.1 `soma_llm_call` slice to
the suite and case that proves it, together with the concrete process-survival
assertion. It is the companion to
[v0.2-test-contract.md](v0.2-test-contract.md),
[v0.3-test-contract.md](v0.3-test-contract.md), and
[v0.4-test-contract.md](v0.4-test-contract.md).

Every actor-facing proof asserts **process survival**, not just a return value:
the worker runs in its own pid, the actor survives the worker's timeout / cancel
/ crash, and the actor stays responsive while a call is in flight.

## What this slice builds

A disposable worker `soma_llm_call` (in `apps/soma_runtime/src/`), shaped like
`soma_tool_call`: it runs in its own process, performs one call, sends a result
message back to its owner, then exits. The actor owns it directly ŌĆö there is no
`soma_llm_call_sup`, mirroring `soma_run ŌåÆ soma_tool_call` (the decision locked
in [v0.4-test-contract.md](v0.4-test-contract.md)).

## Locked design decisions the proofs lean on

Settled in [../../design-77.md](../../design-77.md):

1. **Mutual exclusion** ŌĆö an envelope carries `steps` (list) **or** `llm` (map),
   never both. Both present is malformed and rejected up front by
   `validate_envelope/1` with `{error, _}`, before any child starts.
2. **Owner-enforced timeout** ŌĆö the actor arms a call-timeout timer when it
   starts the call and treats the timer firing as the timeout; the `slow` mock
   that ignores the timer is exactly the case this proves. Mirrors how
   `soma_run` arms the per-step `state_timeout`.
3. **Actor-driven teardown** ŌĆö on timeout and cancel the actor kills the worker
   itself (`exit(WorkerPid, kill)`), because the bare worker is not a
   `gen_statem` that can drive its own teardown. The mock holds no external OS
   process, so there is nothing to reap beyond the BEAM pid.
4. **Events emitted from the actor** ŌĆö `llm.started`, `llm.succeeded`,
   `llm.failed`, `llm.timeout`, `llm.cancelled` are emitted by the actor (which
   already holds the event-store handle), keeping the worker event-store-free.
   Each carries the task's `correlation_id`, so `by_correlation/2` returns them
   alongside the task's `actor.*` events.
5. **Provider seam** ŌĆö the actual call lives in a single function
   `soma_llm_call:perform_call/1`, the one point a real provider grows into. The
   mock reads a `directive` (`success` / `slow` / `crash` / `hang`) out of the
   `llm` map; there is no provider behaviour or `mock` module yet.

## Proving suites

Two suites prove this slice:

- **`soma_llm_call_tests`** ŌĆö an EUnit module in `apps/soma_runtime/test/`. The
  worker-only proof that touches no actor.
- **`soma_llm_call_SUITE`** ŌĆö a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_actor_SUITE` (boot the `soma_runtime` app so `soma_run_sup`
  and the event store are alive, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with an `llm` envelope). Every actor-facing proof.

| 1 | Mock returns configured output, makes no network call | `soma_llm_call_tests` ┬Ę `test_mock_success_returns_configured_output` | `perform_call/1` driven with a `success` directive returns `{ok, Output}`; the "no network call" half is a source-level fact ŌĆö the module opens no socket |
| 2 | Worker runs in a pid distinct from the actor pid | `soma_llm_call_SUITE` ┬Ę `llm_worker_runs_in_distinct_pid` | the worker pid read from the `llm.started` event is a pid and is not the actor pid |
| 3 | `get_task_result` returns the call's output after success | `soma_llm_call_SUITE` ┬Ę `get_task_result_holds_llm_output` | after the success message lands (task `completed`), `get_task_result/2` returns `{ok, Output}` carrying the configured output |
| 4 | Timeout leaves worker dead, task `timeout`, actor alive | `soma_llm_call_SUITE` ┬Ę `slow_call_times_out_worker_dead_actor_alive` | a `slow` mock past a short call timeout: the worker pid is dead (`is_process_alive` false), the task status reads `timeout`, the actor pid is still alive |
| 5 | Cancel leaves worker dead, task `cancelled`, actor alive | `soma_llm_call_SUITE` ┬Ę `cancel_in_flight_call_worker_dead_actor_alive` | cancelling an in-flight `hang` call: the worker pid is dead, the task status reads `cancelled`, the actor pid is still alive |
| 6 | Crash reaches actor via `'DOWN'`, task `failed`, actor alive, and emits `llm.failed` | `soma_llm_call_SUITE` ┬Ę `crash_reaches_actor_as_failed_via_down` | a `crash` mock dies abnormally; the task reaches `failed`, the worker pid is dead, the actor pid is alive and distinct from the dead worker pid; the crash also appends an `llm.failed` event (carrying `correlation_id` + `llm_call_id`) that `by_correlation/2` surfaces beside `actor.task.failed`. | 7 | Actor stays responsive while a call is in flight | `soma_llm_call_SUITE` ┬Ę `status_promptly_while_llm_call_in_flight` | with a `hang` call in flight, `get_task_status/2` returns promptly (well under 200ms) with the non-terminal `running` ŌĆö the actor mailbox is not blocked on the worker |

## v0.5.2 ŌĆö proposals as validated data, not execution

The v0.5.2 slice (see [../../design-83.md](../../design-83.md)) adds a normalized
**proposal** contract: an LLM call returns a raw proposal map, which
`soma_proposal:normalize/1` turns into validated data (`reply` / `run_steps` /
`reject` / `ask`) or rejects. A proposal is *data the actor records* ŌĆö a
`run_steps` proposal does **not** itself start a run. Two suites prove it:

- **`soma_proposal_tests`** ŌĆö an EUnit module in `apps/soma_actor/test/`. The
  pure `soma_proposal:normalize/1` proofs, touching no actor.
- **`soma_proposal_SUITE`** ŌĆö a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_llm_call_SUITE` (boot `soma_runtime`, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with a `proposal` llm directive). The actor-side proofs.

## v0.5.3 ŌĆö policy gate: a proposal gets an allow/reject verdict, no execution

The v0.5.3 slice (see [../../design-85.md](../../design-85.md)) gives a normalized
proposal a **verdict**. A pure `soma_policy:check(Proposal, Policy)` returns
`allow` or `{reject, Reason}` against a tool-name allowlist
(`#{allowed_tools => [atom()] | all}`): a `run_steps` proposal is allowed only
when every step's `tool` is in the allowlist; `all` or an absent `allowed_tools`
key allows any tool; `reply` / `reject` / `ask` carry no tool and are allowed
without a check. The actor wires the gate into the existing `{proposal, Proposal}`
branch right after `proposal.created` and emits `proposal.approved` or
`proposal.rejected` accordingly ŌĆö still **no execution**: an approved `run_steps`
proposal starts no run, it is recorded as approved data. Two suites prove it:

- **`soma_policy_tests`** ŌĆö an EUnit module in `apps/soma_actor/test/`. The pure
  `soma_policy:check/2` proofs, touching no actor.
- **`soma_policy_SUITE`** ŌĆö a Common Test suite in `apps/soma_actor/test/`, set
  up like `soma_proposal_SUITE` (boot `soma_runtime`, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with a proposal-returning `llm` directive). The actor-side gate proofs.

### Pure policy-decision proofs

| # | Proof | Suite ┬Ę case |
|---|-------|--------------|
| 1 | A `run_steps` proposal whose every tool is allowed returns `allow` | `soma_policy_tests` ┬Ę `run_steps_all_tools_allowed_returns_allow_test` |
| 2 | A `run_steps` proposal naming an unknown tool returns `{reject, Reason}` | `soma_policy_tests` ┬Ę `run_steps_unknown_tool_returns_reject_test` |
| 3 | An `all` allowlist (or an absent `allowed_tools` key) returns `allow` | `soma_policy_tests` ┬Ę `run_steps_all_or_absent_allowlist_returns_allow_test` |
| 4 | Toolless kinds (`reply` / `reject` / `ask`) return `allow` without a check | `soma_policy_tests` ┬Ę `toolless_kinds_return_allow_test` |

### Actor-side gate proofs

| # | Proof | Suite ┬Ę case | Survival / behaviour assertion |
|---|-------|--------------|--------------------------------|
| 5 | An allowed `run_steps` proposal emits `proposal.approved` with `correlation_id` | `soma_policy_SUITE` ┬Ę `allowed_run_steps_emits_proposal_approved_with_correlation_id` | `by_correlation/2` surfaces a `proposal.approved` event tagged with the task's `correlation_id` |
| 6 | An allowed proposal starts no run | `soma_policy_SUITE` ┬Ę `allowed_proposal_starts_no_run` | the approved proposal is recorded as data; no `run.*` child is started |
| 7 | An allowed proposal's task status reads `approved` | `soma_policy_SUITE` ┬Ę `allowed_proposal_status_reads_approved` | after the gate, `get_task_status/2` reads `approved`, not `completed` |
| 8 | A rejected proposal emits `proposal.rejected` with the reason and `correlation_id` | `soma_policy_SUITE` ┬Ę `rejected_proposal_emits_proposal_rejected_with_reason_and_correlation_id` | `by_correlation/2` surfaces a `proposal.rejected` event carrying the reject reason and the task's `correlation_id` |
| 9 | A rejected proposal starts no run | `soma_policy_SUITE` ┬Ę `rejected_proposal_starts_no_run` | the rejected proposal starts no `run.*` child |
| 10 | A rejected proposal's task status reads `rejected` | `soma_policy_SUITE` ┬Ę `rejected_proposal_status_reads_rejected` | after the gate, `get_task_status/2` reads `rejected` |
| 11 | The actor survives a rejected proposal and takes the next send | `soma_policy_SUITE` ┬Ę `actor_survives_rejected_proposal_takes_next_send` | after a rejection the actor accepts and completes a second send |
| 12 | `by_correlation/2` returns the verdict, actor and llm events | `soma_policy_SUITE` ┬Ę `by_correlation_returns_verdict_created_actor_and_llm_events` | one correlation id surfaces the verdict (`proposal.approved` / `proposal.rejected`), `proposal.created`, `actor.*` and `llm.*` events together |
