# v0.5 Test Contract â€” soma_llm_call (supervised, monitored LLM-call worker + mock provider)

This document maps each behavioural proof of the v0.5.1 `soma_llm_call` slice to
the suite and case that proves it, together with the concrete process-survival
assertion. It is the companion to
[v0.2-test-contract.md](v0.2-test-contract.md),
[v0.3-test-contract.md](v0.3-test-contract.md), and
[v0.4-test-contract.md](v0.4-test-contract.md).

Every actor-facing proof asserts **process survival**, not just a return value:
the worker runs in its own pid, the actor survives the worker's timeout / cancel
/ crash, and the actor stays responsive while a call is in flight.

## What this slice builds

A disposable worker `soma_llm_call` (in `apps/soma_runtime/src/`), shaped like
`soma_tool_call`: it runs in its own process, performs one call, sends a result
message back to its owner, then exits. The actor owns it directly â€” there is no
`soma_llm_call_sup`, mirroring `soma_run â†’ soma_tool_call` (the decision locked
in [v0.4-test-contract.md](v0.4-test-contract.md)).

## Locked design decisions the proofs lean on

Settled by this v0.5 contract and the suites named below:

1. **Mutual exclusion** â€” an envelope carries `steps` (list) **or** `llm` (map),
   never both. Both present is malformed and rejected up front by
   `validate_envelope/1` with `{error, _}`, before any child starts.
2. **Owner-enforced timeout** â€” the actor arms a call-timeout timer when it
   starts the call and treats the timer firing as the timeout; the `slow` mock
   that ignores the timer is exactly the case this proves. Mirrors how
   `soma_run` arms the per-step `state_timeout`.
3. **Actor-driven teardown** â€” on timeout and cancel the actor kills the worker
   itself (`exit(WorkerPid, kill)`), because the bare worker is not a
   `gen_statem` that can drive its own teardown. The mock holds no external OS
   process, so there is nothing to reap beyond the BEAM pid.
4. **Events emitted from the actor** â€” `llm.started`, `llm.succeeded`,
   `llm.failed`, `llm.timeout`, `llm.cancelled` are emitted by the actor (which
   already holds the event-store handle), keeping the worker event-store-free.
   Each carries the task's `correlation_id`, so `by_correlation/2` returns them
   alongside the task's `actor.*` events.
5. **Provider seam** â€” the actual call lives in a single function
   `soma_llm_call:perform_call/1`, the one point a real provider grows into. The
   mock reads a `directive` (`success` / `slow` / `crash` / `hang`) out of the
   `llm` map; there is no provider behaviour or `mock` module yet.

## v0.5.3 â€” policy gate: a proposal gets an allow/reject verdict, no execution

`allow` or `{reject, Reason}` against a tool-name allowlist

## v0.5.5 â€” budget & loop limits: exhaustion fails the task, not the actor

## v0.5.6 â€” actor-to-actor messages: correlation_id across actors (delivers P12)

The v0.5.6 slice lets an approved
`actor_message` proposal **deliver** to a second actor. One actor (A1) whose mock
returns an approved `actor_message` proposal sends an envelope to a second actor
(A2) whose pid the proposal names; the sender's `correlation_id` rides into A2's
new task, so `by_correlation/2` for that one id returns both actors' events. This
is the slice that **delivers P12** (actor-to-actor messages with a shared
`correlation_id` across actors) â€” the deferred proof carried forward from the
v0.4 contract.

`soma_proposal:normalize/1` gains an `actor_message` success clause (pid `to` +
map `payload`) plus two missing-field failure clauses, all ordered before the
existing `#{kind := Kind}` catch-all. `soma_policy:check/2` gains
`check(#{kind := actor_message}, _) -> allow` (no tools, nothing to gate). The
actor's approved-proposal branch gains an `actor_message` arm next to
`run_steps`: it emits `proposal.executed`, builds a delivery envelope
`#{type => <<"actor.message">>, payload, correlation_id}` carrying the sender
task's `correlation_id`, calls the normal `soma_actor:send(To, Envelope)` entry
point (fire-and-forget, no layer bypassed), then marks the sender task
`completed`. A2 needs no new code â€” its existing `resolve_correlation_id/2` reads
the delivered envelope's `correlation_id`, so A2's task runs under A1's id.

The now-stale `soma_proposal_tests` Â· `test_actor_message_kind_errors` (which
asserted the *kind* is unsupported) is repurposed into a real missing-`payload`
proof, since `actor_message` is now a supported kind.

Three suites/modules prove it:

- **`soma_proposal_tests`** â€” the existing EUnit module in

- **`soma_policy_tests`** â€” the existing EUnit module in
  `apps/soma_actor/test/`. The pure `actor_message` policy-allow proof.
- **`soma_actor_message_SUITE`** â€” a new Common Test suite in
