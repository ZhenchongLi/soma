# v0.5 Test Contract ‚Äî soma_llm_call (supervised, monitored LLM-call worker + mock provider)

This document maps each behavioural proof of the v0.5.1 `soma_llm_call` slice to
the suite and case that proves it, together with the concrete process-survival
assertion. It is the companion to
[v0.2-test-contract.md](v0.2-test-contract.md),
[v0.3-test-contract.md](v0.3-test-contract.md), and
[v0.4-test-contract.md](v0.4-test-contract.md).

Every actor-facing proof asserts **process survival**, not just a return value:
the worker runs in its own pid, the actor survives the worker's timeout / cancel
/ crash, and the actor stays responsive while a call is in flight.

## What this slice builds

A disposable worker `soma_llm_call` (in `apps/soma_runtime/src/`), shaped like
`soma_tool_call`: it runs in its own process, performs one call, sends a result
message back to its owner, then exits. The actor owns it directly ‚Äî there is no
`soma_llm_call_sup`, mirroring `soma_run ‚Üí soma_tool_call` (the decision locked
in [v0.4-test-contract.md](v0.4-test-contract.md)).

## Locked design decisions the proofs lean on

Settled in [../../design-77.md](../../design-77.md):

1. **Mutual exclusion** ‚Äî an envelope carries `steps` (list) **or** `llm` (map),
   never both. Both present is malformed and rejected up front by
   `validate_envelope/1` with `{error, _}`, before any child starts.
2. **Owner-enforced timeout** ‚Äî the actor arms a call-timeout timer when it
   starts the call and treats the timer firing as the timeout; the `slow` mock
   that ignores the timer is exactly the case this proves. Mirrors how
   `soma_run` arms the per-step `state_timeout`.
3. **Actor-driven teardown** ‚Äî on timeout and cancel the actor kills the worker
   itself (`exit(WorkerPid, kill)`), because the bare worker is not a
   `gen_statem` that can drive its own teardown. The mock holds no external OS
   process, so there is nothing to reap beyond the BEAM pid.
4. **Events emitted from the actor** ‚Äî `llm.started`, `llm.succeeded`,
   `llm.failed`, `llm.timeout`, `llm.cancelled` are emitted by the actor (which
   already holds the event-store handle), keeping the worker event-store-free.
   Each carries the task's `correlation_id`, so `by_correlation/2` returns them
   alongside the task's `actor.*` events.
5. **Provider seam** ‚Äî the actual call lives in a single function
   `soma_llm_call:perform_call/1`, the one point a real provider grows into. The
   mock reads a `directive` (`success` / `slow` / `crash` / `hang`) out of the
   `llm` map; there is no provider behaviour or `mock` module yet.

## Proving suites

Two suites prove this slice:

- **`soma_llm_call_tests`** ‚Äî an EUnit module in `apps/soma_runtime/test/`. The
  worker-only proof that touches no actor.
- **`soma_llm_call_SUITE`** ‚Äî a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_actor_SUITE` (boot the `soma_runtime` app so `soma_run_sup`
  and the event store are alive, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with an `llm` envelope). Every actor-facing proof.

## In-scope proofs (v0.5.1)

| # | Proof | Suite ¬∑ case | Survival / behaviour assertion |
|---|-------|--------------|--------------------------------|
| 1 | Mock returns configured output, makes no network call | `soma_llm_call_tests` ¬∑ `test_mock_success_returns_configured_output` | `perform_call/1` driven with a `success` directive returns `{ok, Output}`; the "no network call" half is a source-level fact ‚Äî the module opens no socket |
| 2 | Worker runs in a pid distinct from the actor pid | `soma_llm_call_SUITE` ¬∑ `llm_worker_runs_in_distinct_pid` | the worker pid read from the `llm.started` event is a pid and is not the actor pid |
| 3 | `get_task_result` returns the call's output after success | `soma_llm_call_SUITE` ¬∑ `get_task_result_holds_llm_output` | after the success message lands (task `completed`), `get_task_result/2` returns `{ok, Output}` carrying the configured output |
| 4 | Timeout leaves worker dead, task `timeout`, actor alive | `soma_llm_call_SUITE` ¬∑ `slow_call_times_out_worker_dead_actor_alive` | a `slow` mock past a short call timeout: the worker pid is dead (`is_process_alive` false), the task status reads `timeout`, the actor pid is still alive |
| 5 | Cancel leaves worker dead, task `cancelled`, actor alive | `soma_llm_call_SUITE` ¬∑ `cancel_in_flight_call_worker_dead_actor_alive` | cancelling an in-flight `hang` call: the worker pid is dead, the task status reads `cancelled`, the actor pid is still alive |
| 6 | Crash reaches actor via `'DOWN'`, task `failed`, actor alive, and emits `llm.failed` | `soma_llm_call_SUITE` ¬∑ `crash_reaches_actor_as_failed_via_down` | a `crash` mock dies abnormally; the task reaches `failed`, the worker pid is dead, the actor pid is alive and distinct from the dead worker pid; the crash also appends an `llm.failed` event (carrying `correlation_id` + `llm_call_id`) that `by_correlation/2` surfaces beside `actor.task.failed`. A `soma_run` crash emits no `llm.failed` ‚Äî the shared `'DOWN'` backstop only emits it when the task carries an `llm_call_id` (guarded in `soma_actor_validation_SUITE`) |
| 7 | Actor stays responsive while a call is in flight | `soma_llm_call_SUITE` ¬∑ `status_promptly_while_llm_call_in_flight` | with a `hang` call in flight, `get_task_status/2` returns promptly (well under 200ms) with the non-terminal `running` ‚Äî the actor mailbox is not blocked on the worker |
| 8 | A completed call appends an `llm.*` event with `correlation_id` | `soma_llm_call_SUITE` ¬∑ `completed_call_appends_llm_event_with_correlation_id` | after `completed`, `by_correlation/2` for the envelope's `correlation_id` returns at least one `llm.*`-typed event |
| 9 | `by_correlation/2` returns `llm.*` alongside `actor.*` | `soma_llm_call_SUITE` ¬∑ `by_correlation_returns_llm_and_actor_events` | the same query surfaces both event families for the one task: at least one `actor.*`-typed event AND at least one `llm.*`-typed event under the one `correlation_id` |

## Bonus coverage ‚Äî mutual exclusion (decision 1)

Not a numbered criterion, but load-bearing for the dispatch design, so it gets
its own case.

| Proof | Suite ¬∑ case | Assertion |
|-------|--------------|-----------|
| Both `steps` and `llm` is rejected up front | `soma_llm_call_SUITE` ¬∑ `both_steps_and_llm_rejected_no_child_started` | an envelope carrying both fields returns `{error, _}` from `send/2`; no run and no llm call started; the actor pid is alive |

## v0.5.2 ‚Äî proposals as validated data, not execution

The v0.5.2 slice (see [../../design-83.md](../../design-83.md)) adds a normalized
**proposal** contract: an LLM call returns a raw proposal map, which
`soma_proposal:normalize/1` turns into validated data (`reply` / `run_steps` /
`reject` / `ask`) or rejects. A proposal is *data the actor records* ‚Äî a
`run_steps` proposal does **not** itself start a run. Two suites prove it:

- **`soma_proposal_tests`** ‚Äî an EUnit module in `apps/soma_actor/test/`. The
  pure `soma_proposal:normalize/1` proofs, touching no actor.
- **`soma_proposal_SUITE`** ‚Äî a Common Test suite in `apps/soma_actor/test/`,
  set up like `soma_llm_call_SUITE` (boot `soma_runtime`, start an actor through
  `soma_actor_sup:start_actor/1`, drive it through the real `soma_actor:send/2`
  with a `proposal` llm directive). The actor-side proofs.

### Pure normalize proofs (criteria 1‚Äì8)

|---|-------|--------------|
| 1 | A `reply` proposal normalizes to validated data | `soma_proposal_tests` ¬∑ `test_reply_normalizes_ok` |

| 3 | A `reject` proposal normalizes to validated data | `soma_proposal_tests` ¬∑ `test_reject_normalizes_ok` |
