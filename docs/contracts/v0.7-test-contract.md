## What this slice builds

A durable **resume journal** at the source of truth plus a **read-only
reconstruction API** over the event trail. `soma_run:init/1` emits `run.started`
with a payload of `#{steps => Steps, run_options => DurableRunOptions}`, so the
submitted step list and its resume-safe metadata survive a restart through the
opt-in `disk_log` event store from v0.6. `soma_run_resume:reconstruct/2` reads
that trail back through `soma_event_store:by_run/2` and reports a run's recovered
progress: the journaled steps, the journaled durable options, the committed
outputs keyed by step id, the first uncommitted step, and the terminal status.
Live execution is fenced separately by `soma_run_index`: every attempt claims
its RunId before journalling or invoking a tool, so recovery cannot overlap an
already-live owner from another supervisor generation.

**This slice delivers journaling and read-only reconstruction only — NOT the
execution of resumed runs.** `reconstruct/2` never appends an event, never starts
a `soma_run` child, and never re-runs a tool. `next_step` is a progress marker,
not a permission to resume: a failed, timed-out, or cancelled trail may still
carry an uncommitted step, and deciding whether a resume is allowed is left to a
later resume executor. Starting resumed runs, in-flight idempotency, and actor
task-state reconstruction remain out of scope for v0.7.1.

## Locked design decisions the proofs lean on

1. **Journal at the source of truth.** `soma_run:init/1` is the single place that
   journals resume inputs, covering both production session starts
   (`soma_agent_session:start_run/2`) and direct run starts
   (`soma_run_sup:start_run/1`), because both paths already pass the canonical
   step list into `soma_run`.
2. **Durable run options are allowlist-based.** `DurableRunOptions` carries only
   resume-safe metadata: `run_id` plus optional `session_id`, `correlation_id`,
   `task_id`, `request_id`, `envelope_hash`, `max_output_bytes`,
   `deadline_at_ms`, `auto_resume`, and the fixed allowlisted
   `run_origin = runtime_default | cli_detached`. New ordinary runs journal
   `runtime_default` plus `auto_resume = true`; detached CLI runs journal
   `cli_detached` plus `false`. Unknown origins normalize to an effect-free
   opt-out. It excludes `session_pid`,
   `event_store`, worker pids, monitor refs, timers, OS pids, and any provider
   secret.
3. **The durable log is the source of truth.** A restarted `disk_log` store
   replays the log and serves the journaled `run.started` payload through the
   unchanged `by_run/2` query. Additive finite-timeout `append/3` and `by_run/3`
   variants let recovery owners fail closed during generation changes without
   altering existing caller semantics.
4. **Reconstruction is read-only.** `soma_run_resume:reconstruct/2` calls
   `by_run/2`, derives its result purely, and never calls `append/2`,
   `soma_run_sup:start_run/1`, or any tool execution path. The event store and
   the `soma_run_sup` child tally are unchanged across the call.
5. **A trail must reconcile with its own journal.** Reconstruction tolerates an
   in-progress trail but rejects a trail with no usable `run.started` journal
   (`{error, no_run_started_journal}`) and a trail whose committed step id is
   absent from the journaled steps (`{error, {unknown_committed_step, StepId}}`).
   Committed ids must be one continuous prefix, and the journaled RunId must
   equal the authoritative outer RunId. Malformed tool/args/timeout/reference
   shapes return the fixed bounded diagnostic
   `{error, invalid_run_started_journal}` before any resumed process starts.
6. **Invocation safety is journal-before-effect.** A newly spawned tool worker
   links to its run and waits. `soma_run` durably appends `tool.started` with the
   bounded original `effect`/`idempotent` snapshot before releasing that worker;
   resume requires the original and current safety classifications to agree.
7. **RunId ownership is atomic and generation-fenced.** `soma_run_index`
   monitors one exact live pid per RunId and rejects a second claim. It starts
   before the session/run supervisors. A `soma_run_sup` restart releases old
   claims only after their owners die; an index restart kills and waits for all
   pre-existing runs before serving its new empty table.

## Proving suite

One Common Test suite proves the journal and reconstruction behaviour, and one
EUnit module proves this contract doc:

- **`soma_run_resume_journal_SUITE`** — a CT suite in
  `apps/soma_runtime/test/`. It drives real session starts, direct run starts,
  and a `disk_log` restart through the public runtime API, and exercises
  `soma_run_resume:reconstruct/2` over both driven and synthesized trails.
- **`soma_run_index_SUITE`** — a runtime CT suite that drives real run/index
  claims and isolated supervisor generation changes, including scheduler-
  suspended runs and linked tool workers.
- **`soma_v0_7_contract_doc_tests`** — an EUnit module in
  `apps/soma_runtime/test/` that asserts this contract names the suite and each
  of its cases.

## v0.7.5 — auto-resume on boot

Boot auto-resume adds a durable event-store discovery query and wires it into
runtime application startup when `event_store_log` is configured. The event store
reports replayed run ids whose trail contains `run.started` and no terminal run
event. After `soma_sup` starts, `soma_app` scans those ids and hands each
run only when its durable options contain the exact pair
`run_origin = runtime_default, auto_resume = true`. Every absent, malformed,
unknown, or owner-managed pair fails closed. This intentionally means an
ownership-ambiguous legacy generic journal also requires explicit operator
recovery after upgrade. CLI detached recovery is mapped separately in
[cli-4-test-contract.md](cli-4-test-contract.md).

## Out of scope (recorded for the auditor)

- **Upgrade safety is intentionally conservative.** An upgrade-time in-flight
  trail written before `tool.started.resume_safety` existed fails closed even
  for a reader that is safe in the new registry. Between-step trails and new
  journals are unaffected.

- **No actor task-state reconstruction.** `soma_actor` task recovery is not part
  of this layer.
- **The fail-safe rule is conservative.** A non-idempotent `state` step in flight
  fails rather than re-run; relaxations (a per-tool resume policy, or a
  compensate-then-retry hook) are later work.
- **`next_step` is independent of `terminal_status`.** The plan inspects the
  terminal status before allowing a resume; `next_step` reports progress only.
