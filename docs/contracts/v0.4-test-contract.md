# v0.4 Test Contract ‚Äî soma_actor (agent entity)

This document maps each behavioural proof of the v0.4 `soma_actor` layer to the
slice that delivers it, the planned test suite, and the concrete
process-survival assertion that proves it. It is the companion to
[v0.2-test-contract.md](v0.2-test-contract.md) and
[v0.3-test-contract.md](v0.3-test-contract.md).

It differs from those two in one way, stated up front: v0.2 and v0.3 were
published *after* their code, so their case names are real. **v0.4 is published
first** ‚Äî before any `apps/soma_actor` code ‚Äî because the spec is already
settled in [../zh/soma-actor.zh.md](../zh/soma-actor.zh.md) and
[../design.md](../design.md). So the suite and case names here are **planned**:
each later slice reconciles them as it lands, and a slice may rename a suite
without this doc reading as a broken claim.

The proof list is the `soma_actor` Test Contract in [../design.md](../design.md).
Every proof asserts **process survival**, not just a return value.

## Locked design decisions the proofs lean on

Pinned in the v0.4 umbrella (#52):

1. **New app `apps/soma_actor`** ‚Äî one-way dependency; `soma_runtime` never
   imports it (same boundary as `soma_lfe`).
2. **Direct run start** ‚Äî `soma_actor` calls
   `soma_run_sup:start_run(#{session_pid => self(), ...})` and receives
   `{run_completed | run_failed | run_timeout | run_cancelled, RunId, ...}` in
   its own mailbox. No `soma_agent_session` in the actor's path.
3. **`correlation_id` passthrough** ‚Äî `soma_run` accepts an optional
   `correlation_id` opt and stamps it on every event it emits, so
   `by_correlation/2` recovers run-level events too. Additive, metadata-only.
4. **Single supervisor** ‚Äî `soma_actor_sup` (root, `simple_one_for_one`)
   supervises actor instances. `soma_llm_call` (v0.5) is owner-spawned by
   `soma_actor`, mirroring `soma_run ‚Üí soma_tool_call`; there is no
   `soma_llm_call_sup`.

## In-scope proofs (v0.4)

Twelve proofs: P1‚ÄìP11 and P15.

### Actor lifecycle and task creation

| Proof | Slice | Planned suite | Survival assertion |
|-------|-------|---------------|--------------------|
| P1 ‚Äî actor starts and emits `actor.started` | 3 | `soma_actor_SUITE` | actor pid is alive after `start_link`; an `actor.started` event for `actor_id` is in the store |
| P2 ‚Äî a message creates `task_id` / `correlation_id` | 4 | `soma_actor_SUITE` | `send`/`ask` mints a `task_id`; `actor.message.received` + `actor.task.accepted` carry `task_id` and `correlation_id`; actor pid still alive |

### Run integration

| Proof | Slice | Planned suite | Survival assertion |
|-------|-------|---------------|--------------------|
| P3 ‚Äî runs fixed steps through `soma_run` | 5 | `soma_actor_run_SUITE` | the actor starts a run under `soma_run_sup` (distinct run pid); the demo steps reach `run.completed`; the actor never executes tool logic itself |
| P4 ‚Äî run completion produces an actor result | 5 | `soma_actor_run_SUITE` | on `{run_completed, RunId, Outputs}` the actor records `actor.result.created` + `actor.task.completed`; actor pid still alive |

### Result model

| Proof | Slice | Planned suite | Survival assertion |
|-------|-------|---------------|--------------------|
| P5 ‚Äî `ask` receives a final reply | 6 | `soma_actor_result_SUITE` | `ask(Actor, Envelope, Timeout)` returns `{ok, Result}` for a fixed-step task; caller and actor both alive afterward |
| P6 ‚Äî a long task is queryable by `task_id` | 6 | `soma_actor_result_SUITE` | `get_task_status` / `get_task_result` return the task's status and result for a `task_id` returned by `send` |

### Event trail

| Proof | Slice | Planned suite | Survival assertion |
|-------|-------|---------------|--------------------|
| P7 ‚Äî events queryable by `correlation_id` | 7 | `soma_actor_correlation_SUITE`, `soma_event_store_tests` | `by_correlation/2` returns both `actor.*` and `run.*` events for one `correlation_id` ‚Äî proving decision 3's passthrough reached run-level events |

### Failure isolation and lifecycle

| Proof | Slice | Planned suite | Survival assertion |
|-------|-------|---------------|--------------------|
| P8 ‚Äî actor survives a run failure | 8 | `soma_actor_survival_SUITE` | after `{run_failed, RunId, Reason}` the actor records `actor.task.failed`, its pid is still `is_process_alive`, and it accepts the next envelope |
| P9 ‚Äî actor survives a tool crash | 8 | `soma_actor_survival_SUITE` | a crashing tool fails the run (monitor `'DOWN'` inside `soma_run`); the failure reaches the actor as data, not a crash signal; actor pid still alive |
| P15 ‚Äî actor stays responsive while a child run is active | 8 | `soma_actor_survival_SUITE` | while a run is mid-flight, `get_status` returns promptly ‚Äî the actor is not blocked on the child |
| P10 ‚Äî cancel task cancels the active run | 9 | `soma_actor_lifecycle_SUITE` | `cancel(Actor, TaskId)` stops the active `soma_run` (run reaches `cancelled`, its tool worker is gone); the actor records `actor.task.cancelled`; actor pid still alive |
| P11 ‚Äî actor accepts another message after failure / cancel / timeout | 9 | `soma_actor_lifecycle_SUITE` | after each terminal outcome the actor accepts a fresh envelope and completes a new task |

## Deferred to v0.5

Three proofs from the design's contract are out of scope for v0.4's fixed-rule
slice, because each needs the LLM-call / structured-proposal machinery that v0.4
does not build:

| Proof | Why deferred |
|-------|--------------|
| P12 ‚Äî actor-to-actor message preserves `correlation_id` | Needs two actors with a real reason to message each other. Structurally it is `send/2` to another actor pid with `correlation_id` propagating, but there is nothing meaningful to prove it against until v0.5. |
| P13 ‚Äî budget exhaustion fails the task, not the actor | Needs the budget / loop-limit accounting that gates LLM calls and runs. |
| P14 ‚Äî policy rejection fails or asks, not the actor | Needs the policy gate over a structured proposal, which only exists once `soma_llm_call` produces proposals. |

The seam: when v0.5 adds `soma_llm_call`, `soma_actor` spawns and monitors it
directly (mirroring `soma_run ‚Üí soma_tool_call`), and the proposal it returns
flows through the policy gate before execution.
