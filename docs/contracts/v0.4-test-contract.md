# v0.4 Test Contract — soma_actor (agent entity)

This document maps each behavioural proof of the v0.4 `soma_actor` layer to the
slice that delivers it, the test suite and case(s) that prove it, and the
concrete process-survival assertion. It is the companion to
[v0.2-test-contract.md](v0.2-test-contract.md) and
[v0.3-test-contract.md](v0.3-test-contract.md).

It differs from those two in one way, noted for context: v0.2 and v0.3 were
published *after* their code. **v0.4 was published first** — this file landed
before any `apps/soma_actor` code, because the spec was already settled in
[../zh/soma-actor.zh.md](../zh/soma-actor.zh.md) and
[../design.md](../design.md). Now that all nine slices have merged, the suite
and case names below are **reconciled to the real tests**. All twelve in-scope
proofs are green on `main`; nearly every case lives in `soma_actor_SUITE`, with
P7 in `soma_actor_correlation_SUITE`.

The proof list is the `soma_actor` Test Contract in [../design.md](../design.md).
Every proof asserts **process survival**, not just a return value.

## Locked design decisions the proofs lean on

Pinned in the v0.4 umbrella (#52):

1. **New app `apps/soma_actor`** — one-way dependency; `soma_runtime` never
   imports it (same boundary as `soma_lfe`).
2. **Direct run start** — `soma_actor` calls
   `soma_run_sup:start_run(#{session_pid => self(), ...})` and receives
   `{run_completed | run_failed | run_timeout | run_cancelled, RunId, ...}` in
   its own mailbox. No `soma_agent_session` in the actor's path.
3. **`correlation_id` passthrough** — `soma_run` accepts an optional
   `correlation_id` opt and stamps it on every event it emits, so
   `by_correlation/2` recovers run-level events too. Additive, metadata-only.
4. **Single supervisor** — `soma_actor_sup` (root, `simple_one_for_one`)
   supervises actor instances. `soma_llm_call` (v0.5) is owner-spawned by
   `soma_actor`, mirroring `soma_run → soma_tool_call`; there is no
   `soma_llm_call_sup`.

## In-scope proofs (v0.4)

Twelve proofs: P1–P11 and P15. Suite names are abbreviated below: **AS** =
`soma_actor_SUITE`, **CS** = `soma_actor_correlation_SUITE`.

### Actor lifecycle and task creation

| Proof | Slice | Suite · cases | Survival assertion |
|-------|-------|---------------|--------------------|
| P1 — actor starts and emits `actor.started` | 3 | AS · `start_emits_one_actor_started_event`, `actor_started_event_carries_actor_id`, `actor_alive_after_start` | actor pid is alive after `start_actor`; an `actor.started` event for `actor_id` is in the store |
| P2 — a message creates `task_id` / `correlation_id` | 4 | AS · `send_mints_task_id_when_absent`, `correlation_id_defaults_to_task_id`, `message_received_event_carries_ids`, `task_accepted_event_matches_received_ids` | `send` mints a `task_id`; `actor.message.received` + `actor.task.accepted` carry `task_id` and `correlation_id`; actor pid still alive |

### Run integration

| Proof | Slice | Suite · cases | Survival assertion |
|-------|-------|---------------|--------------------|
| P3 — runs fixed steps through `soma_run` | 5 | AS · `run_started_under_run_sup_distinct_pid`, `run_completes_with_run_event_trail`, `actor_run_worker_pids_all_distinct` | the actor starts a run under `soma_run_sup` (distinct run pid); the demo steps reach `run.completed`; actor, run, and tool-call pids are all distinct — the actor never executes tool logic itself |
| P4 — run completion produces an actor result | 5 | AS · `result_created_event_carries_ids`, `task_completed_event_carries_ids`, `task_status_completed_after_run`, `task_result_holds_outputs_after_run` | on `{run_completed, RunId, Outputs}` the actor records `actor.result.created` + `actor.task.completed`, stores the outputs as the task result; actor pid still alive |

### Result model

| Proof | Slice | Suite · cases | Survival assertion |
|-------|-------|---------------|--------------------|
| P5 — `ask` receives a final reply | 6 | AS · `ask_returns_run_outputs`, `ask_reply_matches_completed_run`, `ask_caller_and_actor_alive_after_return` | `ask(Actor, Envelope, Timeout)` returns `{ok, Result}` for a fixed-step task only after the run completes; caller and actor both alive afterward |
| P6 — a long task is queryable by `task_id` | 6 | AS · `get_task_status_queryable_by_send_task_id`, `get_task_status_completed_after_run`, `get_task_result_ok_outputs_after_completion` | `get_task_status` / `get_task_result` return the task's status and result for a `task_id` returned by `send` |

### Event trail

| Proof | Slice | Suite · cases | Survival assertion |
|-------|-------|---------------|--------------------|
| P7 — events queryable by `correlation_id` | 7 | CS · `test_chain_retrievable_by_correlation_id` | `by_correlation/2` returns both `actor.*` and `run.*` events for one `correlation_id` — proving decision 3's passthrough reached run-level events |

### Failure isolation and lifecycle

| Proof | Slice | Suite · cases | Survival assertion |
|-------|-------|---------------|--------------------|
| P8 — actor survives a run failure | 8 | AS · `failed_run_emits_task_failed_event`, `failed_run_sets_task_status_failed`, `actor_alive_after_owned_run_fails`, `ask_failed_run_returns_error` (and the timeout variant `timed_out_run_emits_task_failed_timeout`, `ask_timed_out_run_returns_error_timeout`) | after `{run_failed, RunId, Reason}` the actor records `actor.task.failed`, its pid is still `is_process_alive`, a parked `ask` gets `{error, Reason}`, and it accepts the next envelope |
| P9 — actor survives a tool crash | 8 | AS · `tool_crash_isolated_by_process_boundary` | a crashing tool fails the run (monitor `'DOWN'` inside `soma_run`); the failure reaches the actor as data, not a crash signal; actor pid still alive, distinct from the dead run/worker pids |
| P15 — actor stays responsive while a child run is active | 8 | AS · `status_running_promptly_while_run_in_flight`, `read_returns_while_earlier_run_in_flight` | while a run is mid-flight, `get_task_status` returns promptly with `running` — the actor is not blocked on the child |
| P10 — cancel task cancels the active run | 9 | AS · `cancel_drives_run_to_cancelled`, `cancel_kills_tool_call_worker`, `cancel_emits_actor_task_cancelled_event`, `cancel_status_cancelled_and_actor_alive` | `cancel(Actor, TaskId)` stops the active `soma_run` (run reaches `cancelled`, its tool worker is verifiably dead); the actor records `actor.task.cancelled`; actor pid still alive |
| P11 — actor accepts another message after failure / cancel / timeout | 9 | AS · `new_run_completes_after_failed_run`, `new_run_completes_after_timed_out_run`, `new_run_completes_after_cancelled_run` | after each terminal outcome the actor accepts a fresh envelope and completes a new task |

## Deferred to v0.5

Three proofs from the design's contract are out of scope for v0.4's fixed-rule
slice, because each needs the LLM-call / structured-proposal machinery that v0.4
does not build:

| Proof | Why deferred |
|-------|--------------|
| P12 — actor-to-actor message preserves `correlation_id` | Needs two actors with a real reason to message each other. Structurally it is `send/2` to another actor pid with `correlation_id` propagating, but there is nothing meaningful to prove it against until v0.5. |
| P13 — budget exhaustion fails the task, not the actor | Needs the budget / loop-limit accounting that gates LLM calls and runs. |
| P14 — policy rejection fails or asks, not the actor | Needs the policy gate over a structured proposal, which only exists once `soma_llm_call` produces proposals. |

The seam: when v0.5 adds `soma_llm_call`, `soma_actor` spawns and monitors it
directly (mirroring `soma_run → soma_tool_call`), and the proposal it returns
flows through the policy gate before execution.
