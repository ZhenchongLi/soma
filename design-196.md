## Current state

The repository already preserves the core OTP boundaries described in `README.md` and `AGENTS.md`: sessions start and observe runs, runs own step state, tool calls cross a `soma_tool_call` process boundary, actors own LLM-call workers, and failures are supposed to become terminal run/task data rather than crashing owners.

The issue targets five edge paths where the current implementation does not yet enforce that boundary:

- `apps/soma_actor/src/soma_cli_main.erl` builds the auto-start command in `launch_daemon/1` with string concatenation and `os:cmd/1`: `nohup "<self>" daemon --socket "<sock>" ... &`. Quoting does not make arbitrary socket paths shell-safe, so a `--socket` value containing shell metacharacters can be interpreted by the shell.
- `apps/soma_actor/src/soma_cli_server.erl` has `unlink_stale/1` delete any filesystem entry at the socket path if a probe connect fails. That correctly clears a stale AF_UNIX socket, but it also deletes an ordinary file at that path.
- `apps/soma_actor/src/soma_actor.erl` arms an LLM timeout only when the envelope/config supplies `timeout_ms`; `arm_llm_timeout(undefined, _) -> undefined`. A hanging provider or mock call without an explicit timeout can keep the task running forever.
- `apps/soma_runtime/src/soma_llm_openai.erl` calls `httpc:request/4` with an empty HTTP option list, leaving the live provider path on `httpc`'s unbounded default request timeout.
- `apps/soma_runtime/src/soma_agent_session.erl` accepts only list-shaped `Steps` at the public API guard, but it does not validate each step before starting a run. `soma_run:executing/3` then calls `maps:get(id, Step)` and `maps:get(tool, Step)`, so a non-map or a map missing `id`/`tool` can crash the run instead of producing a named validation failure.
- `apps/soma_runtime/src/soma_tool_call.erl` forwards the raw result of an in-BEAM `Module:invoke/2` to `soma_run`. `soma_run:waiting_tool/3` only handles `{ok, _}` and `{error, _}` result shapes. Any other return leaves the run stuck in `waiting_tool` unless the step has an explicit timeout.

Existing test locations match these seams: CLI daemon tests under `apps/soma_actor/test/`, actor LLM timeout tests in `soma_llm_call_SUITE`, OpenAI provider unit tests in `soma_llm_openai_tests`, and runtime failure tests in `soma_run_failure_SUITE`.

## Approach

Keep the implementation narrow and boundary-oriented. Do not change the supervision shape, the Lisp wire protocol, the CLI command surface, or the sequential step-list contract.

For daemon auto-start, prefer a shell-free detached launch over path rejection. Add a small helper around `open_port({spawn_executable, Self}, ...)` in `soma_cli_main:launch_daemon/1`, passing `["daemon", "--socket", Sock]` as argv and using port options such as `nouse_stdio`, `stderr_to_stdout`, `exit_status`, and `hide` where available. The client does not need to hold the port until the daemon exits; it only needs to initiate the separate OS process and return `ok`, preserving the existing best-effort `ensure_daemon/2` behavior. If portability of full detachment is awkward, the fallback may reject unsafe socket paths, but the preferred design is to remove the shell interpolation surface entirely.

For socket cleanup, make `soma_cli_server:unlink_stale/1` type-aware. Use `file:read_file_info/1`; leave `#file_info{type = regular}` and other non-socket ordinary file types untouched. For the AF_UNIX leftovers used by the supported macOS/Linux targets, treat `#file_info{type = other}` as a candidate stale socket: probe it with `gen_tcp:connect({local, Path}, ...)`; if the probe succeeds, close it and leave the path alone; if the probe fails, delete it. Preserve `{error, _} -> ok`.

For actor-owned LLM timeouts, introduce a fixed module constant such as `?DEFAULT_LLM_TIMEOUT_MS` in `soma_actor`. Change `arm_llm_timeout(undefined, LlmCallId)` to start a timer with that default, while preserving explicit positive integer `timeout_ms` as the override. The timeout result should use the existing timeout path: kill the worker, emit `llm.timeout`, record task status `timeout`, release any waiter, and leave the actor alive.

For the OpenAI-compatible provider, add a bounded request timeout to the live `httpc` path. Keep the fixed `response` seam socket-free. In `chat/1`, pass an HTTP option like `[{timeout, RequestTimeoutMs}]` to `httpc:request/4`, where `RequestTimeoutMs` comes from `maps:get(request_timeout_ms, Config, ?DEFAULT_OPENAI_REQUEST_TIMEOUT_MS)` or reuses `timeout_ms` if the implementation wants one timeout key. Keep the error normalization unchanged: `{error, Reason}` becomes `{error, {http_request_failed, Reason}}`. Do not emit config or API key material.

For malformed runtime steps, validate before a run can crash. The cleanest boundary is `soma_agent_session:handle_call({start_run, Steps}, ...)`: validate that `Steps` is a list and that every element is a map carrying `id` and `tool`. On failure, append the existing `run.accepted` only if the design intentionally treats validation as a run attempt; otherwise return a named `{error, {invalid_step, Detail}}` without starting `soma_run`. Because the acceptance criterion says "Starting a run through the runtime/session entry ... fails ... and the session stays alive", the recommended behavior is to still return `{ok, RunId}`, emit `run.accepted` and `run.failed` with `#{reason => {invalid_step, Detail}}`, record status `failed`, and start no `soma_run` child. That preserves the public run-status/event contract while avoiding a doomed process. If the team prefers `{error, Reason}`, tests should assert that shape explicitly and verify no run child starts; in either case the reason must be named and the session must remain alive.

For invalid in-BEAM tool returns, normalize inside `soma_tool_call` before replying. Wrap only the returned value shape, not tool execution broadly: `{ok, _}` and `{error, _}` pass through unchanged; any other term becomes `{error, {invalid_tool_return, Term}}` or `{error, invalid_tool_return}` if the term may be too large. The run will then take its existing `{error, Reason}` path and record `tool.failed`, `step.failed`, and `run.failed`. This keeps the worker boundary meaningful and avoids teaching `soma_run` about arbitrary callback shapes.

## Acceptance criteria → tests

- Shell metacharacters in daemon auto-start socket path do not execute an embedded command: add an EUnit case in `apps/soma_actor/test/soma_cli_main_tests.erl` that sets `SOMA_SELF` to a harmless test executable/script, calls the auto-start path through `soma_cli_main:dispatch(["status", "missing", "--socket", MaliciousPath])` or a direct exported/test seam if one exists, and asserts the marker file named in the injected command is absent after a short wait.
- Regular file at stale socket path is not deleted: add a Common Test case to `apps/soma_actor/test/soma_cli_server_SUITE.erl` that writes a regular file at `Path`, calls `soma_cli_server:start_link(#{socket => Path})`, expects an error or non-bind outcome, and asserts `file:read_file(Path)` still returns the original bytes.
- Stale AF_UNIX socket file is still deleted: keep and tighten the existing `test_start_link_unlinks_stale_socket_file/1` in `soma_cli_server_SUITE`. Create a real AF_UNIX socket path by starting a server and stopping/killing it so the leftover file has socket/`other` type, then assert a new `start_link/1` on the same path succeeds and accepts a connection. Avoid using a regular file for this proof after the cleanup behavior changes.
- Actor LLM call without explicit `timeout_ms` gets default owner timeout: add a case to `apps/soma_actor/test/soma_llm_call_SUITE.erl` using `Llm = #{directive => hang}` or `#{directive => slow}` with no timeout. Assert the task reaches `timeout`, the worker pid from `llm.started` is dead, and the actor pid remains alive.
- `openai_compat` uses bounded HTTP request timeout: add a structural or seam-based EUnit proof in `apps/soma_runtime/test/soma_llm_openai_tests.erl`. Preferred: factor request options into a pure exported helper such as `request_http_options/1` and assert it includes `{timeout, N}` with `N` a positive integer by default and honors an override. This avoids opening a network socket in the gate.
- Malformed non-map runtime step fails with named validation and session alive: add a case in `apps/soma_runtime/test/soma_run_failure_SUITE.erl` starting through `soma_agent_session:start_run(SessionPid, [not_a_map])`. Assert the public result/event/status shape chosen above carries a named validation reason and `is_process_alive(SessionPid)` is true.
- Malformed map missing `id` or `tool` fails with named validation and session alive: add two sibling cases in `soma_run_failure_SUITE`, one for `#{tool => echo, args => #{}}` and one for `#{id => s1, args => #{}}`, asserting the named reason distinguishes missing `id` / missing `tool` enough to debug and the session stays alive.
- Invalid in-BEAM tool return fails boundedly: add a test-only tool module under the runtime test support path or extend the existing `fail` tool only if that does not weaken its contract. Register a manifest for e.g. `bad_return` with `adapter => erlang_module`, `module => soma_tool_bad_return`, start a run that invokes it without `timeout_ms`, and assert `run.failed` arrives promptly, the failure payload names `invalid_tool_return`, no `run.timeout` is needed, and the session remains alive.

## Risks & trade-offs

- Shell-free daemon launch may have subtle release/runtime differences. Keep it isolated in `launch_daemon/1`, keep the existing `SOMA_SELF=false/"" -> ok` behavior, and test only the security outcome rather than process-manager internals.
- File type reporting for AF_UNIX paths is platform-sensitive. The issue's supported targets expect stale socket paths to appear as `other`; the implementation should leave regular files untouched and limit deletion to the socket-like type(s) proven on macOS/Linux.
- Adding a default LLM timeout changes behavior for tasks that previously could run forever. That is intentional for this hardening issue, but choose a conservative fixed value so real provider calls are bounded without surprising normal use.
- Validating malformed steps at the session boundary creates a choice between returning `{error, Reason}` and recording a failed run id. The event/status contract is stronger if malformed step lists become failed run data with a run id, but the implementation must keep that behavior explicit in tests.
- Including the invalid callback return term in `{invalid_tool_return, Term}` is helpful but can leak large data into events. If bounded event size is a concern, store only the tag and possibly the term's type/class.
