## Real issues

- The production loop is unbounded unless every caller supplies valid limits. `soma_delegate_request:valid_field_shapes/1` only checks that `budgets` is a map at `apps/soma_actor/src/soma_delegate_request.erl:52`, `counter_available/3` treats every missing counter cap as unlimited at `apps/soma_actor/src/soma_delegate_coordinator.erl:1242`, and `arm_task_deadline/2` treats an invalid deadline as no deadline at `apps/soma_actor/src/soma_delegate_coordinator.erl:883`. A one-entry fixed action request with no budgets consumed 24 round-worker and LLM-call starts before manual cancellation; `deadline_ms => <<"forever">>` also normalizes successfully. Reject incomplete or ill-typed adaptive budgets, or install bounded defaults, before coordinator creation.

- One action can run several non-idempotent state steps, but the safety ledger records only the first. `first_unsafe_invocation/2` stops at the first unsafe descriptor at `apps/soma_actor/src/soma_delegate_round_worker.erl:778`, and `maybe_add_adaptive_mutation/4` commits that single map at line 684. A two-step production probe completed with `tool_calls => 2` and only `state_one` under `mutations`; `state_two` vanished from the next prompt, audit state, idempotency state, and terminal projection. Give every state invocation its own identity and ledger outcome.

- A malformed proposal map can crash the round worker instead of returning bounded malformed task data. `admit_raw_proposal/2` calls `soma_proposal:normalize/1` at `apps/soma_actor/src/soma_delegate_round_worker.erl:421`, but `normalize(#{})` has no matching clause. The task eventually reports `failed` through worker-loss handling, while the supervisor logs a crash and no `delegate.decision.completed` event exists. Make normalization total for external model data and emit the invalid decision before terminalization.

- Coordinator loss destroys the task's safety trail. `soma_delegate:remove_active_coordinator/3` fabricates `coordinator_crashed_projection()` from ingress route data at `apps/soma_actor/src/soma_delegate.erl:224`; that route owns no counters, mutation ledger, unknown-outcome ledger, or unsafe-dispatch marker. Killing the coordinator after a non-idempotent `tool.started` event produced `status => failed`, zero usage, empty safety ledgers, and a terminal event containing only `phase`, `status`, and `usage_count`. This must become an honest `in_doubt` outcome with preserved safety state, and the terminal event must keep the adaptive schema.

- The terminal projection fallback does not enforce its own byte cap. `bounded_terminal_projection/2` checks once at `apps/soma_actor/src/soma_delegate.erl:528`, then `overflow_terminal_projection/1` retains unbounded `usage` and never checks the replacement at line 565. A valid provider reply with a 12,000-digit `prompt_tokens` count returned a 5,290-byte public projection and replaced the requested `<<"done">>` result with `terminal_projection_too_large`. Bound provider usage and prove the fallback fits without rewriting a valid output-contract result.
