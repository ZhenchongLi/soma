## Real issues

- The mutation ledger records the selected plan, not the tools that actually ran. `unsafe_invocations/2` allocates identities for every unsafe step at `apps/soma_actor/src/soma_delegate_round_worker.erl:813`, and `maybe_add_adaptive_mutation/4` stamps the run's aggregate outcome onto all of them at line 718. A production probe whose first state step returned a known error emitted one `tool.started` event but returned two `failed` mutations, including the state step that never started. The opposite failure exists on cancellation: `run_cancelled` reports no mutation at line 306, and the coordinator commits that empty result at `apps/soma_actor/src/soma_delegate_coordinator.erl:489`; cancelling after a non-idempotent `tool.started` event returned `cancelled` with empty mutation and unknown-outcome ledgers. Build safety state from actual per-step dispatch and terminal facts. An unresolved started state call must remain `in_doubt`.

- Valid provider usage totals collapse to zero at the public boundary. Each OpenAI response may report up to `16#ffffffff` prompt tokens at `apps/soma_runtime/src/soma_llm_openai.erl:12`, and the coordinator correctly sums completed calls at `apps/soma_actor/src/soma_delegate_coordinator.erl:628`. The ingress rejects any aggregate above that same per-call limit at `apps/soma_actor/src/soma_delegate.erl:722` and replaces the whole usage map with zeros at line 649. Two fixed responses reporting 3,000,000,000 prompt tokens each under `max_total_prompt_tokens => 7000000000` completed successfully, then exposed four zero counters. Use one compatible bound from request admission through provider parsing, task totals, and terminal projection.

- A valid observation budget can silently discard a completed action. The worker keeps output inline whenever it fits caller-supplied `max_observation_bytes` at `apps/soma_actor/src/soma_delegate_round_worker.erl:769`, but the coordinator rejects every round result above its independent 16,384-byte cap at `apps/soma_actor/src/soma_delegate_coordinator.erl:1000` and ignores it at line 438. With `max_observation_bytes => 20000`, a 17,000-byte `echo` completed through `tool.succeeded` and `run.completed`; the next prompt had no recent round, no `delegate.action.completed` event existed, and a later reply still made the task `succeeded`. Artifactize before the round envelope cap, or reject incompatible observation limits at ingress.
