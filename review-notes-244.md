## Real issues

- `apps/soma_actor/src/soma_actor_sup.erl:29` starts `soma_service`, whose `init/1` immediately replays durable runs, before `apps/soma_actor/src/soma_actor_app.erl:12` registers the production `ask_actor` descriptor. A crash-window trail containing `service.task.accepted` and `run.started` but no `tool.started` resumed on ten out of ten fresh boots, then landed `tool.failed {unregistered_tool, ask_actor}` before the application registered the tool. The step never crossed the tool boundary before the crash, so restart-safe pending work is being discarded. Register actor-owned descriptors before service recovery or add a registry-ready gate, then pin this boot trail.
- `apps/soma_actor/src/soma_service.erl:182` copies `soma_policy`'s full `tools_not_allowed` list into the public task and durable `service.task.terminal` event. A valid 100,000-step rejected envelope produced a 600,177-byte reply and a 600,078-byte durable payload. Repeating one disallowed tool makes rejection data grow with the plan, violating the repository's bounded task/event rule and letting rejected work inflate the durable log. Collapse or cap the policy detail before storing it, and add an encoded-size pin.
