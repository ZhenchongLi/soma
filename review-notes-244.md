## Real issues

- `apps/soma_actor/src/soma_service.erl:295` feeds every positive normalized deadline into `erlang:start_timer/3`. `deadline_ms => 1 bsl 100` passes normalization, then the timer raises `badarg`. The accepted task is already journaled and its slow run is live. The replacement service crashes on the same value at line 616, reaches `max_restart_intensity`, and takes down `soma_actor`. The probe left both the run and tool worker alive under `soma_runtime`. One valid envelope defeats permanent ownership. Reject timer-unsafe values before journaling or arm long deadlines in bounded chunks. Recovery must turn existing poison metadata into bounded terminal data instead of crashing again.
- `apps/soma_actor/src/soma_service.erl:646` reconstructs a terminal run before enforcing the persisted absolute deadline. Only an existing `service.task.deadline_expired` marker wins at line 756. A trail with a past `deadline_at_ms`, then `step.succeeded` and `run.completed`, but no marker returned `succeeded` with the late output. This is the service-down crash window: work completed after expiry is reported as success. Recovery must compare the deadline with the ordered durable outcome and must not resume already-expired safe work.
