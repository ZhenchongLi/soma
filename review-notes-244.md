## Real issues

- `apps/soma_actor/src/soma_service.erl:554` honors a durable deadline marker only when the reconstructed run status is `cancelled`. A trail containing `service.task.deadline_expired`, `step.succeeded`, and `run.completed` returned `succeeded` after restart. The deadline decision must win over every later run outcome and every adopted-run message. Restore that owner decision before live adoption or trail reconstruction, then cover the crash window after the marker append.
- `apps/soma_actor/src/soma_service.erl:426` cannot replay the success that `recover_committed_outputs/3` records for an all-steps-committed trail without `run.completed`. The first restart returned the exact successful result. The second restart changed the same task to `failed service_result_reconstruction_failed`. Recover recorded success from committed outputs too, and prove the terminal survives another service restart.
- `apps/soma_actor/src/soma_service.erl:72` publishes a failed task when the monitored run dies, but the dead run was the only owner of its tool worker and external process. Killing an active sleep run left its `soma_tool_call` process alive after the service exposed `failed`. A CLI worker keeps its port and child program in the same state. Make invocation resources die with the run or clean them before publishing the task terminal.
- `apps/soma_actor/src/soma_service.erl:79` copies the raw run exit reason into the public task and `service.task.terminal`. A forced 20,000-byte exit reason produced a 20,169-byte task and the same unbounded durable payload. Normalize this branch to bounded task data. Do not persist raw crash terms.
- `apps/soma_actor/src/soma_service.erl:374` calls `last_terminal_event/2` over the complete event list once per accepted task. Successful task recovery adds another full-store scan through `soma_run_resume:reconstruct/2`. Startup is quadratic: replaying 1,000 terminal tasks took 36.4 ms; 2,000 took 137.7 ms in the same VM. Build the task, terminal, and run-trail lookup maps in one pass so the permanent service can restart against a growing durable log.
